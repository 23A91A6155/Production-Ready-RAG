Retrieval-Augmented Generation (RAG): A Technical Overview
===========================================================

Introduction
------------
Retrieval-Augmented Generation (RAG) is a machine learning architecture that combines information retrieval with large language model (LLM) text generation. It was introduced by Lewis et al. in 2020 in a paper titled "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks." The core idea is to enhance LLMs by grounding their answers in external, retrieved documents rather than relying purely on parametric knowledge stored in model weights.

Why RAG Was Created
-------------------
Traditional LLMs are trained on massive text corpora and encode knowledge into their weights. While powerful, this approach has significant limitations:
- Knowledge cutoff: Models cannot access information beyond their training date.
- Hallucination: Models often generate plausible-sounding but factually incorrect statements.
- Lack of citations: Answers cannot be traced back to specific sources.
- Expensive updates: Re-training or fine-tuning a model to update its knowledge is expensive.

RAG addresses all of these problems by coupling retrieval with generation at inference time.

Core Components of a RAG System
--------------------------------
A complete RAG pipeline consists of two major phases: the Ingestion Phase and the Query Phase.

Ingestion Phase:
1. Document Loading - Raw text files (.txt, .md, .pdf) are read from disk and cleaned.
2. Text Chunking - Documents are split into smaller pieces called chunks. Chunking is necessary because embedding models and LLM context windows have token limits. Common strategies include fixed-size chunking with overlap to preserve cross-boundary context.
3. Embedding Generation - Each chunk is passed through an embedding model (such as OpenAI's text-embedding-3-small) to produce a dense vector representation. This vector captures the semantic meaning of the chunk.
4. Vector Storage - The chunk embeddings, along with the raw text and metadata, are stored in a vector database such as ChromaDB or FAISS. The data is persisted locally so ingestion only needs to run once.

Query Phase:
1. Query Embedding - The user's question is converted into a vector using the same embedding model used during ingestion. This is critical: using a different model would produce incompatible vector spaces.
2. Similarity Search - The query vector is compared against all stored chunk vectors using cosine similarity. The top-K most similar chunks are retrieved.
3. Context Construction - Retrieved chunks are assembled into a context block, including their source metadata.
4. Prompt Construction - A structured prompt is built that contains: system instructions, the retrieved context, and the user question. The prompt explicitly instructs the LLM to answer only from the provided context.
5. LLM Generation - The prompt is sent to an LLM such as GPT-4o-mini. The model generates an answer grounded in the context.
6. Answer and Citations - The final output includes the generated answer plus the source chunks and document names that were used.

Chunking Strategies
--------------------
Chunking is one of the most important design decisions in a RAG system. The chunk size controls the granularity of retrieval. Smaller chunks are more precise but may lack context; larger chunks provide more context but may retrieve irrelevant information. A chunk size of 300 to 500 tokens is commonly used in production RAG systems.

Overlap is used to avoid losing context at chunk boundaries. If a key sentence spans the end of one chunk and the beginning of the next, overlap ensures that at least one chunk captures it fully. A typical overlap is 10 to 20 percent of the chunk size.

Embedding Models
-----------------
The embedding model is responsible for converting text into numerical vectors. OpenAI's text-embedding-3-small model produces 1536-dimensional vectors and is optimized for retrieval tasks. The critical rule is that the same model must be used for both ingestion (chunk embedding) and query (question embedding). Using different models would create vectors in different semantic spaces, making similarity search meaningless.

Vector Databases
-----------------
Vector databases are purpose-built for storing and querying high-dimensional embedding vectors. ChromaDB is a popular open-source option that supports local persistence and cosine similarity search out of the box. FAISS (Facebook AI Similarity Search) is another option optimized for speed at large scale. Both support the cosine similarity metric, which measures the angle between two vectors and is more stable than Euclidean distance for high-dimensional spaces.

Cosine Similarity
------------------
Cosine similarity is computed as the dot product of two unit-normalized vectors. A cosine similarity of 1.0 means the vectors are identical in direction; 0.0 means orthogonal (unrelated); -1.0 means opposite. In practice, cosine similarity values between a query and relevant chunks typically fall in the range 0.7 to 0.95.

Prompt Engineering in RAG
--------------------------
The RAG prompt is carefully designed to constrain the LLM to use only retrieved context. A well-structured RAG prompt includes:
- A system instruction telling the model to answer only from the provided context.
- The context block with numbered chunks and source citations.
- The user question.
- An explicit fallback instruction: if the answer is not in the context, respond with "I don't know."

This design prevents hallucination by removing the model's incentive to fabricate answers.

Limitations of RAG
-------------------
Despite its advantages, RAG has known limitations:
- Retrieval quality depends heavily on chunk quality and embedding model capability.
- If the relevant information is not in the ingested documents, the system correctly returns "I don't know" but cannot help further.
- Long documents may require many chunks, increasing retrieval latency.
- Overlap increases storage requirements but is generally worth the cost.

Applications of RAG
--------------------
RAG is widely used in enterprise knowledge bases, customer support systems, legal and medical document Q&A, code documentation assistants, and research tools. Any domain where accurate, source-grounded answers are critical benefits from RAG over pure LLM generation.

Conclusion
----------
RAG represents a foundational shift in how LLMs are deployed for knowledge-intensive tasks. By separating knowledge storage (vector database) from knowledge processing (LLM), RAG systems are updatable, auditable, and significantly less prone to hallucination than purely parametric models.
